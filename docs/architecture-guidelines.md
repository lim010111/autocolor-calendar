# Architectural Guidelines & Conventions

- **Source of Truth:** Incremental sync (`nextSyncToken`) is the authoritative source of truth. Webhooks should only be treated as wake-up signals, not as complete data records.
- **Sync Flow:** The reliable flow is:
  1. Wake up from a Calendar webhook (Watch API) or scheduled run on the backend.
  2. Call the Calendar API with an incremental sync token or equivalent stored checkpoint.
  3. Fetch the actual changed events.
  4. Re-evaluate color rules for those events.
  5. Update event colors when needed.
  6. Save the new sync state.
- **Idempotency:** Color updates should be idempotent to avoid infinite loops when the app modifies its own calendar events.
- **Color Ownership (§5.4):** The app marks every event it colors with three keys under Google Calendar `extendedProperties.private`: `autocolor_v` (schema version, currently `"1"`), `autocolor_color` (the colorId we wrote), and `autocolor_category` (the categoryId that drove the choice). On each subsequent sync, an event is considered **app-owned and re-applicable** only if `event.colorId === extendedProperties.private.autocolor_color` — i.e., the marker color still matches the current color. If the marker is missing, or the user changed the color after our last PATCH (current colorId !== stored autocolor_color), the event is treated as user-manual and skipped (`SyncSummary.skipped_manual`). The app never retro-claims a color it didn't write — when `current === target` and the marker is absent, we record `skipped_equal` without stamping a marker. Deleting a category (`DELETE /api/categories/:id`) fans out `color_rollback` queue jobs (one per calendar in `sync_state`) whose consumer clears the color override + all three markers on events whose `autocolor_category` still matches — but only when the ownership check passes, so user re-painted events remain untouched. See `src/CLAUDE.md` "Color ownership marker" for the full contract.
- **Workspace Add-on Constraints:** The main Add-on UI MUST be built using `CardService` in the `gas/` directory. Complex configuration UIs can be implemented via `HTMLService` or external Web UIs linked from the cards.
- **Domain Verification:** Real-time sync via Webhooks (Watch API) requires a verified domain via Google Search Console. Ensure the production endpoint is properly registered.
- **Watch renewal concurrency (§6.4 / §4B M4):** `renewExpiringWatches` guards each `stop → register` round with a per-row claim on `sync_state.watch_renewal_in_progress_at` (10-min TTL, ms-precision ownership probe). This column is deliberately separate from `sync_state.in_progress_at` (sync consumer claim) — sync and renewal touch independent Google API surfaces, so a single lock would pointlessly serialise them. Any future manual-trigger route MUST reuse `claimWatchRenewal` / `releaseWatchRenewal` against this column; see `src/CLAUDE.md` "Watch renewal concurrency (§6.4)" for the full contract.
- **E2E Backend Mandatory:** All users MUST complete backend (Supabase/Workers) authentication during onboarding to use the service. Local GAS Triggers for syncing or coloring are **DEPRECATED and MUST NOT BE USED**.
- **Halt on Failure:** If the backend fails or API communication errors occur, processing must halt. There is NO fallback to local rules. (OAuth `invalid_grant` is the narrow exception: the refresh token is revoked, so the failure path must surface a re-login prompt to the user rather than silently retrying.)
- **Hybrid Classification Engine:** The backend utilizes a **2-step pipeline**: Stage 1 **embedding kNN** (Supabase pgvector) → PII redaction → LLM Fallback. Stage 1 was the substring keyword matcher; ADR-0004 #02 replaced it with embedding kNN (see [`docs/adr/0004-embedding-classifier.md`](adr/0004-embedding-classifier.md) + [`docs/adr/0005-embedding-model-eval-ko-v1.md`](adr/0005-embedding-model-eval-ko-v1.md), and `src/CLAUDE.md` §5.1). Stage 1 embeds the event title (Workers AI, model + prefix frozen in `src/config/embedding.ts`) and takes the max cosine over the user's `rule_seeds` vectors; `score(best) < T_declared` or `best - second < margin` (or an embedding failure) hands off to the LLM leg, otherwise it assigns the best rule. Dimension (768) and thresholds are ADR-0005 **provisional**. PII redaction is mandatory and non-bypassable before any LLM call (see `src/CLAUDE.md` log-redaction contract and §5.2 of `TODO.md`); the Stage-1 embedding call is a Cloudflare-internal Workers AI call that does not cross the PII boundary. On LLM failure there is NO fallback to local rules — the event is left with `no_match` and skipped silently (consistent with the "Halt on Failure" bullet above). Timeouts, network errors, 4xx/5xx responses, malformed LLM output, unknown category names, per-user OR global daily quota exhaustion, and missing `OPENAI_API_KEY` all collapse identically into `no_match`; per-outcome telemetry lives in `SyncSummary.llm_*` counters, and the `llm_calls` table (§6 Wave A) records per-call `outcome` / `latency_ms` / `category_count` / `attempts` / `category_name` for dashboards. The same pipeline and daily quota are re-used by the on-demand sidebar preview (`POST /api/classify/preview { llm: true }`) — preview writes one `llm_calls` row per request via `execCtx.waitUntil` and shares `llm_usage_daily` with sync, so there is no separate preview cap. Preview is additionally rate-limited via `users.last_preview_at` (2-second minimum interval) — see `src/CLAUDE.md` "Preview LLM (§5 후속)" + "Cost guardrail (§5/§6 후속)" for the full contract.
- **Cost guardrail (§5/§6 후속):** Operator-side OpenAI-credit protection layered on top of the §5.3 per-user 200/day cap. (1) **Global daily call cap** (`LLM_GLOBAL_DAILY_LIMIT`, default 10,000/day, env-overridable) via `llm_usage_global_daily` UPSERT in `reserveLlmCall`, ordered global-first so global exhaustion does not consume per-user budget. The cross-user UPSERT is the **second documented exception** to the "Tenant isolation" rule below (the first is the §3-후속 token-rotation cron). (2) **Preview throttle** via `users.last_preview_at` — 2s minimum interval, atomic UPDATE-RETURNING, sole-writer column following the `sync_state.last_manual_trigger_at` (§6.4) discipline. (3) **Prompt-side input caps** in `buildPrompt` (summary 256 / description 1024 / location 256 UTF-16 chars) — the sync path's actual chokepoint, since the route-layer Zod cap protects only `/api/classify/preview`. (4) **Response-token cap** via `max_completion_tokens: 64` — overruns truncate to `bad_response` → silent miss. (5) **Daily cost report cron** `5 0 * * *` UTC summarising `llm_usage_global_daily` to `console.warn` at 80% saturation, info-level otherwise; failure-isolated with `.catch(warn)` so observability never drives cron retry. See `src/CLAUDE.md` "Cost guardrail (§5/§6 후속)" for the full contract.
- **Failure audit tables (§6 Wave A):** Three persistent surfaces carry observability data: `sync_failures.summary_snapshot` (`SyncSummary` copied from `sync_state.last_failure_summary` at DLQ-write time), `llm_calls` (per-LLM-call outcome/latency log, bulk-inserted at sync-run end), and `rollback_runs` (per-`color_rollback`-job audit, one row per outcome including retryable attempts). Writers never fail the underlying job: `llm_calls` uses `execCtx.waitUntil(...).catch(warn)` and `rollback_runs` uses `await db.insert(...).catch(warn)` — observability failure downgrades to a warn log rather than triggering a retry that would re-run side-effectful work. See `src/CLAUDE.md` "Observability tables" for the full contract.
- **LLM per-event debugging surface (§6.3 후속):** `llm_calls` carries four nullable columns (`event_id` / `prompt_summary` / `raw_response` / `available_categories`) populated by `classifyWithLlm`'s `finish()` helper across all 7 outcome branches — see `src/CLAUDE.md` "Observability tables (§6 Wave A)" for the per-outcome NULL-policy matrix. `prompt_summary` inherits the §5.3 redaction contract (`redactEventForLlm` runs upstream, only `summary`/`description`/`location` cross the prompt boundary), so adding these columns introduces no new PII surface — they are DB columns, not log lines, and the "Calendar event payloads must never be logged" rule binds the log stream only. Self-query reader: `GET /api/llm-calls?window=24h|7d|30d&outcome=...&eventId=...&limit=...` (`src/routes/llmCalls.ts`) — tenant-scoped via `where(eq(llmCalls.userId, ctx.userId))`. This surface replaces a deferred Langfuse integration: Langfuse JS v5 SDK packages all declare `engines.node >= 20` and depend on `@opentelemetry/sdk-node`, with no docs-blessed Cloudflare Workers path. Re-evaluate Langfuse when (a) prompt version management, (b) LLM-as-judge auto-scoring, or (c) Workers-blessed Langfuse SDK becomes a need.
- **Reader surface (§6 Wave B):** `sync_runs` (per-`runPagedList`-invocation audit) joins the Wave A trio as the fourth observability table and is the historical source that `GET /api/stats?window=7d|30d` aggregates. One row per Worker invocation across all six `RunResult` outcomes (`ok` / `reauth_required` / `not_found` / `forbidden` / `full_sync_required` / `retryable`), guaranteed by a `finalize(result)` helper that every `return` inside `runPagedList` passes through. Same fire-and-forget discipline: `syncConsumer.handleOne` injects `recordSyncRun` as `execCtx.waitUntil(db.insert(syncRuns).values(...).catch(warn))` so an observability write failure never triggers `msg.retry`. `/api/stats` performs four parallel SELECTs (`sync_runs` + `llm_calls` + `rollback_runs` + today's `llm_usage_daily` row) through Hyperdrive — GAS `buildHomeCard` consumes the result to replace the "이번 주 분류된 일정" mock with live counters. `sync_state.last_run_summary` is intentionally preserved as the inline-snapshot surface for `/me`. See `src/CLAUDE.md` "Observability tables (§6 Wave B)" for the full contract.
- **User-initiated deletion (§3 row 179):** `POST /api/account/delete` is the user-facing privacy gate. `DELETE FROM users` is the sole authoritative writer; FK cascade fans out to 9 user-scoped tables (oauth_tokens / sessions / categories / sync_state / llm_usage_daily / sync_failures / llm_calls / rollback_runs / sync_runs). Google API cleanup (refresh-token revoke + `channels.stop` per active watch row) runs **before** the DELETE so cascade-dropped rows are still readable, and is best-effort — failures log warn-only and never block the authoritative delete (narrow exception to "Halt on Failure" above, justified because the user has explicitly asked to leave). Idempotency is provided by the auth gate, not the route: a second call carries a bearer that no longer resolves a session → 401. See `src/CLAUDE.md` "Account deletion (§3 row 179)" for the full contract including the deliberate "observed, not prevented" stance toward in-flight sync / watch-renewal claims.
- **Token encryption rotation invariant (§3 후속):** `TOKEN_ENCRYPTION_KEY_PREV` is an env binding that MUST exist **only** during an active rotation window. It is the sole input to the rotation cron's PREV-key decrypt path (`src/services/tokenRotation.ts` → `rotateBatch`) and to the `getGoogleRefreshToken` dual-key fallback. Outside the window the binding MUST be removed (`wrangler secret delete TOKEN_ENCRYPTION_KEY_PREV --env <target>`) — leaving it set after the cron drains the `token_version <> TARGET_TOKEN_VERSION` set to zero is a security regression because it widens the key surface area without any data still depending on it. The rotation cron's cross-user `SELECT` against `oauth_tokens` (filtering on `token_version` only, with no `userId` predicate) is the ONE documented exception to the "Tenant isolation" rule in `src/CLAUDE.md`; every other `oauth_tokens` query MUST keep its `where(eq(oauthTokens.userId, ...))` clause. Per-row failures inside the rotation batch are warn-only — never rethrown — so observability writes can't trigger cron retry, consistent with the `llm_calls` / `rollback_runs` / `sync_runs` discipline. See `src/CLAUDE.md` "Token rotation (§3 후속)" for the writer/reader contract and the operator runbook in "Secret rotation impact".

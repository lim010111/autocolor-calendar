  - **해결**: `docs/assets/marketplace/reviewer-demo/05-rule-deletion-rollback.md` 신규. 슬라이스 1·2·3·4와 동일 골격(Scopes 콜아웃 → Pre-conditions → "Two consent surfaces" pre-read → 단계별 `## N.` → Failure modes → Cross-references). 4단계 walkthrough — (1) reviewer가 슬라이스 2에서 만든 `주간회의` 카테고리의 삭제 버튼을 규칙 관리 카드에서 클릭 → GAS가 `DELETE /api/categories/:id` 호출. (2) backend `src/routes/categories.ts` DELETE 핸들러가 (a) tenant-scoped DELETE 후 (b) `sync_state`의 모든 calendarId를 SELECT해 calendar당 `color_rollback` queue 잡 fan-out (`COLOR_ROLLBACK_QUEUE.send`). (3) queue consumer (`src/services/colorRollback.ts`)가 `events.list?privateExtendedProperty=autocolor_category=<id>` 페이징으로 후보 이벤트 수집 → §5.4 ownership 검증(`event.colorId === extendedProperties.private.autocolor_color`) 통과 이벤트만 `clearEventColor` PATCH (colorId: null + 3 marker keys all null). 사용자가 PATCH 이후 색을 직접 바꾼 이벤트는 **검증 실패 → skip** (rollback marker 무효화로 reviewer가 직접 확인 가능). (4) 각 잡의 outcome (`ok` / `reauth_required` / `forbidden` / `not_found` / `retryable`)이 `rollback_runs` 테이블에 1행 insert (fire-and-forget, attempt=msg.attempts). reviewer는 Calendar 그리드에서 rollback 대상 이벤트가 default color로 복귀, 수동 변경 이벤트는 보존되는 것을 직접 확인. Failure modes 5개: (a) AUTH_EXPIRED → `reauth_required` outcome + `buildReconnectCard`, (b) calendar permission 회수 → `forbidden`, (c) calendar 자체 삭제 → `not_found`, (d) transient 5xx → `retryable` (재시도 후 max attempts 도달 시 DLQ 적재 + `rollback_runs.outcome="retryable"` + `attempt=msg.attempts` 기록), (e) 사용자 수동 변경 이벤트 (ownership check 실패) → 그 이벤트만 skip, queue 잡 자체는 `ok`. **`docs/marketplace-readiness.md` 행 5 status `미작성` → `초안`** + Source of truth pointer `src/services/colorRollback.ts` → `docs/assets/marketplace/reviewer-demo/05-rule-deletion-rollback.md`, README scenario matrix 행 5 mirror 동기화.

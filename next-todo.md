- **§6.4 Watch 갱신 동시성 가드 — `watch_channels.in_progress_at` 컬럼 + claim/release 헬퍼 (§4B 리뷰 M4)**
  - **문제**: `renewExpiringWatches`는 Cloudflare cron으로 주기 실행되지만, 향후 운영 대시보드(§6.3 Wave B 이후)에서 "지금 갱신" 수동 트리거 경로가 생기면 cron과 overlap될 수 있다. 현재 service는 채널별로 순차 stop→register를 수행하는데, 두 Worker invocation이 같은 expiring row를 동시에 집으면 A가 stop 직후 register한 새 채널을 B가 "만료 임박"으로 오판해 또 stop해버리는 race가 가능하다. §4B M4에서 "row-level `in_progress_at` 스탬프 또는 짧은 dedup window 도입 검토"로 이월된 상태.
  - **해결**: §4A Finding #2 / §6.4 `last_manual_trigger_at`과 동일한 **row-level `in_progress_at`** 패턴을 채택. `drizzle/<next>` 마이그레이션에 `watch_channels.in_progress_at timestamptz null` + 보조 btree 인덱스 추가. `src/lib/syncClaim.ts`의 `claimSyncRun`/`releaseSyncRun`을 레퍼런스로 `src/lib/watchClaim.ts` 신설 — `date_trunc('milliseconds', now())`로 ms 정밀도 일치시키고, 10분 TTL(기존 release가 없으면 stale로 간주 후 탈취 가능)로 stuck row 복구 경로 확보. `renewExpiringWatches`의 per-channel 루프 안쪽에서 `claimRenewal → try stop+register finally releaseRenewal`로 감싼다. claim 실패(이미 누군가 작업 중) 시 해당 row 건너뜀 — 다음 cron이 자연 재처리.
  - **주요 변경**: (1) `drizzle/<next>_watch_channels_in_progress.sql` — ALTER TABLE + btree `(in_progress_at)` 인덱스 + RLS 주석. (2) `src/db/schema.ts` `watchChannels`에 `inProgressAt` 필드 추가. (3) `src/lib/watchClaim.ts` 신설 — `claimRenewal(db, channelId, ttlMs)` / `releaseRenewal(db, channelId, claimedAt)` 2개 export, `syncClaim.ts` 시그니처 미러링. (4) `src/services/watchChannels.ts` (또는 `watchRenewal.ts`) 의 renewal 루프에 claim/release 래핑 추가. (5) 테스트: `src/__tests__/watchRenewal.test.ts`에 concurrency 가드 describe 3~4 케이스 (claim 성공 / claim 실패 → skip / release 이후 재-claim 가능 / stale TTL 탈취). (6) `src/CLAUDE.md`에 "Watch 갱신 동시성 (§6.4 / §4B M4)" 섹션 추가 — `in_progress_at`의 정본 writer와 consumer 미스-라이트 금지 invariant 명시.
  - **문서**: `src/CLAUDE.md` "Watch 갱신 동시성" 섹션 신설. `docs/architecture-guidelines.md` Watch 관련 불릿(있다면) 보완 또는 cross-ref 추가. `TODO.md` §6.4 해당 체크박스 flip.
  - **의존성**: 없음. `drizzle-kit generate`로 로컬 마이그레이션 생성 후 `pnpm db:migrate`로 dev Supabase 적용 (prod 적용은 §3 후속 "Prod 환경 활성화"와 묶어 진행).

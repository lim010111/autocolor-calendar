OAuth Consent Screen 검수 reviewer가 "토큰이 revoke되면 앱이 어떻게 행동하는가"를 확인할 때 직접 인용되는 행.

  - **해결**: `docs/assets/marketplace/reviewer-demo/06-reauth-invalid-grant.md` 신규. 슬라이스 1·2·3·4·5와 동일 골격(Scopes 콜아웃 → Pre-conditions → "Two consent surfaces" pre-read → 단계별 `## N.` → Failure modes → Cross-references). 4단계 walkthrough — (1) reviewer가 Google 계정 보안 설정에서 AutoColor 접근을 revoke (또는 `oauth_tokens.encrypted_refresh_token`을 직접 망가뜨려 `invalid_grant` 시뮬레이션). (2) reviewer가 AutoColor 사이드바에서 임의 동작 (예: `actionSyncNow` 클릭) → `gas/api.js`의 `fetchBackend`가 backend `/sync/run` 호출. (3) backend `getValidAccessToken` → `src/services/tokenRefresh.ts`의 refresh 시도가 Google에서 `errorCode === "invalid_grant"` 응답을 받음 → `markReauthRequired(userId)`로 `oauth_tokens.needs_reauth = true` 플립 + `ReauthRequiredError` throw → Worker가 401 반환. (4) GAS 사이드바: `actionSyncNow`의 catch가 `err.message === 'AUTH_EXPIRED'` 검출 → `buildReconnectCard()` (`gas/addon.js:1107`)로 navigation pop → reviewer가 `"Google 다시 연결"` 버튼을 통해 OAuth 재시작 → 성공 시 `oauth_tokens`이 새 refresh token으로 갱신되고 `needs_reauth` 클리어. Failure modes 5개: (a) `invalid_grant`가 watch renewal cron 중에 surface (`watchRenewal.ts`도 동일 `markReauthRequired` 경로 — user-visible card는 다음 사이드바 호출 때), (b) `invalid_grant`가 `color_rollback` consumer 중에 surface (`reauth_required` outcome으로 `rollback_runs` 1행 + 동일 reauth flag), (c) `invalid_grant`가 sync consumer mid-run에 surface (`reauth_required` outcome 으로 `sync_runs` 1행), (d) `gas/authError.html` `?error=<code>` 중 `invalid_grant` 외 코드 (state_invalid / consent_denied / provider_error / token_exchange_failed / server_error) — 각각 한국어 메시지 분기, (e) reviewer가 reconnect 카드의 `"Google 다시 연결"` 버튼을 누르지 않고 OAuth를 외부에서 다시 grant한 경우 — 다음 사이드바 호출 때 `oauth_tokens.needs_reauth` 플래그가 여전히 set이므로 `/me`가 `{ needs_reauth: true }` 반환 → 카드 유지. **`docs/marketplace-readiness.md` 행 6 status `미작성` → `초안`** + Source of truth pointer `gas/authError.html` → `docs/assets/marketplace/reviewer-demo/06-reauth-invalid-grant.md`, README scenario matrix 행 6 mirror 동기화.
